# 剩余问题

这些问题不是完全不能回答，而是当前代码还没有把口径做实，或者需要你我先统一实验叙述再写进 README/论文。

## 1. `cost` 到底应不应该拆成 token 和 latency

当前实现里，utility 用的是归一化步成本，不是真实 token 或真实 latency。

这会带来一个问题：

- 决策时优化的是“步数代价”
- 评测时报告的是“token / wall_time”

这两者之间并不完全一致。

可以讨论的改法：

1. 保持当前写法，把它明确命名为 `step_cost`
2. 改成 `expected token cost`
3. 改成 `expected latency cost`
4. 拆成双成本：
   `U = gain - l1 * token_cost - l2 * latency_cost - ...`

我更建议至少把名字改成 `step_cost`，避免读者误以为这里真的是 token cost。

## 2. `expected_gain` 和 `uncertainty` 是否需要更严格定义

当前两者都是 LLM self-estimation。

这在工程上能跑，但论文里会被追问：

- 是否校准过
- 是否能复现
- 是否和最终 F1 真相关

可以讨论的替代方案：

1. 保留 self-estimation，并明确说明是 heuristic signal
2. 用 verifier / reward model 估计
3. 用 sampled answer disagreement 做 uncertainty
4. 用 retrieval improvement proxy 做 expected_gain

如果你想往论文更严谨的方向走，我建议后面至少加一个“小节：这些量是 heuristic，不声称是 calibrated probability”。

## 3. `redundancy` 现在太弱，要不要升级

当前 redundancy 只是“query 是否完全重复”，这个定义很弱。

会漏掉很多近义重复，例如：

- `who founded x`
- `x founder`

它们语义重复，但当前会被当成不重复。

可以讨论的升级方案：

1. BM25 top-k overlap
2. query embedding cosine similarity
3. evidence passage overlap
4. n-gram Jaccard

如果你要发 paper，我建议至少升级到 “retrieved passage overlap” 或 “embedding similarity”。

## 4. workflow baseline 是否足够公平

这是 README 里最值得认真处理的问题之一。

当前 workflow 就是一个固定 baseline：

- search once
- then answer

它的好处是简单、可复现、容易解释。

但你的担心是对的：

- 如果 baseline 设计太弱，会显得对比不公平

这里建议我们一起定一个写法：

1. 把当前 workflow 明确写成“minimal fixed pipeline baseline”
2. 增加 1~2 个额外 fixed workflow baseline
3. 或者在文中承认它不是 exhaustive workflow family，只是最小固定流程对照

我更建议选第 2 种。

## 5. policy 为什么优于 threshold 的“实验角度”

设计角度现在能讲，但实验角度必须等结果出来后再写。

至少需要看：

- `avg_f1`
- `avg_tokens`
- `avg_wall_time`
- `redundant_tool_calls`
- `stop_step_distribution`

只有这些结果出来后，才能判断：

- 是质量更高
- 还是同质量更省
- 还是只是多搜了几次

所以这一条建议等实验跑完再补。

## 6. 是否要把“不是全局最优”写进 README

当前 policy 是局部贪心，不是全局最优，也不是 A*。

这件事我建议直接写明，不要模糊处理。因为如果不写，读者一追问就会暴露。

更稳妥的表述可以是：

`Our policy is a myopic utility-based controller rather than a globally optimal planner.`

## 7. README / 论文中建议统一的几个术语

当前代码和 README 里有些术语容易混：

- `cost`
- `step_cost`
- `token_usage`
- `wall_time`
- `uncertainty`
- `expected_gain`

我建议后续统一成下面这套：

- `step_cost`：policy 内部使用的归一化步成本
- `token_usage`：实验统计里的 token 开销
- `wall_time`：实验统计里的延迟
- `uncertainty`：LLM self-estimated uncertainty
- `expected_gain`：LLM self-estimated marginal value of one more search

这样读者不容易把决策变量和评测指标混起来。
