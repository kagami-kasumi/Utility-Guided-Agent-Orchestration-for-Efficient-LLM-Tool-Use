# 实验汇总

本文档基于当前仓库里的实际实现整理，不是理想化设计稿。下面的回答描述的是“代码现在到底做了什么”。

## 1. 为什么 `lambda` 是小数，不是整数

当前 utility function 里的几个量都被压在接近 `0~1` 的尺度上：

- `expected_gain`：模型输出后被裁剪到 `[0, 1]`
- `uncertainty`：模型输出后被裁剪到 `[0, 1]`
- `redundancy`：当前是二值 `0/1`
- `cost`：当前是按步数归一化后的 `step_cost`

在这个尺度下，`lambda` 用小数更容易平衡各项贡献。如果直接用整数，很容易让某一项直接压过其它项，导致 policy 过于激进或过于保守。

当前默认值：

- `lambda_cost = 0.3`
- `lambda_uncertainty = 0.2`
- `lambda_redundancy = 0.2`

## 2. 后面 3 个数值是怎么定义的

这里指的是 `lambda_cost / lambda_uncertainty / lambda_redundancy`。

- `lambda_cost`：惩罚继续调用工具的代价
- `lambda_uncertainty`：惩罚当前回答不确定性
- `lambda_redundancy`：惩罚重复检索

当前 utility 写法：

```text
U(a|s) =
expected_gain
- lambda_cost * cost
- lambda_uncertainty * uncertainty
- lambda_redundancy * redundancy
```

## 3. `expected_gain` 是怎么计算的

当前不是外部显式公式计算，也不是用验证集监督学习出来的。现在的实现是：

- 让模型根据当前 observation 自己输出一个 `expected_gain`
- 读取这个 JSON 字段
- 再裁剪到 `[0,1]`

如果模型没给合法值，就用默认值 `0.4`。

## 4. `cost` 是怎么估计的

当前 `cost` 不是 token，也不是 latency。现在代码里使用的是归一化步成本：

```text
step_cost = (current_tool_calls + 1) / max_tool_calls
```

含义是：越接近预算上限，再继续搜的代价越高。

所以当前 policy 里的 `cost` 更接近“继续搜索的机会成本”，而不是系统层面的真实计算成本。

## 6. `uncertainty` 怎么定义

当前 `uncertainty` 也是模型自报，不是 entropy、variance 或校准后的概率。

实现方式：

- prompt 让模型返回 `uncertainty`
- 程序将其裁剪到 `[0,1]`
- 如果缺失，默认值是 `0.5`

## 7. `redundancy` 怎么定义

当前定义比较简单，不是 embedding 相似度，也不是 passage overlap。

现在的实现：

- 把 query 转成小写并去首尾空格
- 如果当前 query 和历史 query 中某个完全相同，则 `redundancy = 1.0`
- 否则 `redundancy = 0.0`

所以它本质上是“重复 query 检测”。

## 8. 是否做了 normalization

做了部分 normalization：

- `expected_gain`：clip 到 `[0,1]`
- `uncertainty`：clip 到 `[0,1]`
- `cost`：按 `max_tool_calls` 归一化
- `redundancy`：二值 `0/1`

没有做的：

- 没把真实 `tokens` 归一化后放进 utility
- 没把真实 `latency` 归一化后放进 utility
- 没做 uncertainty / expected_gain 的 calibration

## 10. `lambda` 是怎么选的

当前不是训练学出来的，也不是理论推导出来的。

当前分两层：

- 主实验默认先用一组手工 baseline：`(0.3, 0.2, 0.2)`
- 再通过 utility ablation/超参数实验去扫一组候选值

所以当前更准确的表述是：

`lambda` 目前是人工初始化 + 网格搜索验证稳定性。

## 12. 每一步是不是都在使 utility 最大，是不是贪心

是的，当前实现是局部贪心决策，不是全局最优搜索。

更准确地说：

- 每一轮根据当前 state 计算一次 utility
- 如果 `utility > 0`，且没超过 `max_tool_calls`，就继续 search
- 否则停下来输出答案

它不是 A*，也没有做 lookahead、beam search 或 dynamic programming，因此不能保证全局最优。

## 13. 为什么说 policy 在设计上可能优于 threshold

只能先回答“设计上可能为什么更强”，不能替代最终实验结论。

当前设计差异：

- `threshold`：只看 `uncertainty * expected_gain - cost_weight * cost`
- `policy`：显式拆成 `expected_gain - lambda_cost * cost - lambda_uncertainty * uncertainty - lambda_redundancy * redundancy`

这意味着 `policy` 比 `threshold` 多了两个灵活性：

- 可以单独控制不确定性和成本的权重
- 可以显式抑制重复检索

所以设计上 `policy` 的表达能力更强，更容易做 ablation 和解释。
